### 1.2 架构风格

> 基于教学场景和3小时开发时间约束，选择轻量级、易理解的架构风格。

| 决策 | 选型 | 理由 |
|------|------|------|
| **前后端分离** | React SPA + Flask REST API | 职责分离清晰，前端独立开发服务器，教学演示方便 |
| **单体部署** | 前后端各一个进程 | 不引入微服务复杂度，适合3小时快速开发 |
| **分层架构** | Route → Agent → Storage | 替换存储引擎时上层零改动，符合教学分层思想 |
| **简化LLM策略** | 百炼 → Demo 两级降级 | 时间限制下优先保证核心功能可用，CoPaw集成复杂度较高暂缓 |
| **本地文件存储** | JSON 文件存储 | 零配置部署，无需数据库安装，适合教学和演示 |
| **状态管理** | React Hooks (useState/useEffect) | 组件级状态管理足够，避免引入Redux等复杂状态库 |

## 2. 后端分层结构

> 每层写清"允许做什么、禁止做什么"。

| 层 | 文件 | 允许做 | 禁止做 |
|----|------|--------|--------|
| **Route** | `routes/*.py` | 参数校验、HTTP 状态码、traceId、统一响应包装 | 直接读写文件、调用 LLM、业务逻辑处理 |
| **Agent** | `agents/*.py` | 业务编排、LLM调用协调、结果组装、溯源信息构建 | 感知 HTTP 请求/响应、直接操作存储 |
| **Service** | `services/*.py` | 导出文件生成、数据转换、格式化 | 调用外部 API、直接读写存储 |
| **Storage** | `storage.py` | 文件 CRUD、数据校验、JSON序列化 | 调用外部 API、业务逻辑处理 |

## 3. 前端架构

| 项 | 选型 | 理由 |
|----|------|------|
| UI 框架 | React（Hooks）≥ 18 | 教学通用性最高，组件化思想清晰 |
| 构建工具 | Vite ≥ 5 | 极快 HMR，零配置启动，3小时内快速迭代 |
| 状态管理 | `useState` + `useEffect` + Context | 会话状态、用户信息等全局状态用Context，局部用Hooks |
| HTTP 客户端 | `fetch` API | 浏览器原生，不引入 axios，减少依赖 |
| 样式方案 | CSS Modules + CSS Variables | 样式隔离避免冲突，支持主题变量统一管理；配合Vite原生支持 |
| UI组件库 | Ant Design 或 自研简单组件 | 快速构建专业投研界面，或根据时间选择自研 |
| 路由 | React Router | 登录页、问答页、研报管理页路由切换 |

## 5. 数据存储架构

### 5.1 存储演进路线

| 阶段 | 存储 | 变更范围 | 说明 |
|------|------|----------|------|
| **P0（当前）** | JSON 文件 | — | 适合3小时快速开发，单文件存储会话、问答、研报元数据 |
| **P1** | SQLite 本地数据库 | Storage 层实现类替换 | 支持事务、索引、并发访问，数据量增大时性能更好 |
| **P2** | PostgreSQL + Redis | Storage 层 + 配置 + 部署架构 | 生产级关系型数据库支持复杂查询；Redis缓存热点研报分析结果 |

### 5.2 数据文件结构

```
data/
├── sessions.json          # 会话列表
├── qa_records.json        # 问答记录
├── reports.json           # 研报元数据
├── report_contents/       # 研报解析内容
│   ├── report_001.json
│   └── report_002.json
└── exports/               # 导出文件临时存储
    └── export_001.txt
```

## 7. ADR（架构决策记录）

> 每条 ADR 记录一个"为什么选 A 而不选 B"的决策。

| ADR | 决策 | 核心理由 |
|-----|------|----------|
| **ADR-001** | Flask 而非 FastAPI | 学习曲线低，同步代码更易理解，适合教学场景和3小时快速开发 |
| **ADR-002** | JSON文件存储而非数据库（P0阶段） | **背景**：P0阶段仅有3小时开发时间，需要快速启动和部署，降低环境配置复杂度。<br>**决策**：采用JSON文件作为数据存储方案。<br>**理由**：<br>1. 零配置部署：无需安装和配置数据库服务，学员可直接运行<br>2. 教学友好：数据文件可直接查看和编辑，便于理解数据模型<br>3. 快速原型：适合MVP阶段快速验证功能逻辑<br>4. 时间约束：3小时内无法完成数据库设计、迁移、ORM配置<br>**后果**：数据量大时性能下降，并发写入存在竞态条件；无事务支持<br>**替代方案**：SQLite（P1采用，单文件但仍需SQL语法）、PostgreSQL（P2采用，完整关系型数据库） |
| **ADR-003** | 百炼+Demo两级降级而非三级 | **背景**：时间限制3小时，需优先保证核心功能可用。<br>**决策**：采用百炼→Demo两级降级策略，暂缓CoPaw集成。<br>**理由**：<br>1. 时间效率：CoPaw需要内部环境配置和对接开发，百炼API可直接调用<br>2. 功能完整：百炼提供稳定的LLM能力，Demo兜底保证离线可用<br>3. 可扩展性：架构预留CoPaw接入点，后续可无缝扩展为三级<br>4. 教学聚焦：让学员关注降级逻辑本身，而非多供应商适配<br>**后果**：失去CoPaw内部数据对接能力；百炼调用产生API成本<br>**替代方案**：三级降级（CoPaw→百炼→Demo）：功能更完整但开发时间不足；单一LLM：无降级保障，服务不可用时时系统崩溃 |
| **ADR-004** | 本地JSON存储研报内容而非向量数据库 | **背景**：需要支持研报解析和问答溯源，但时间受限。<br>**决策**：采用本地JSON文件存储解析后的研报结构化数据。<br>**理由**：<br>1. 简化架构：无需引入向量数据库（如Milvus、Pinecone）<br>2. 快速实现：JSON可直接映射Python字典，无需ORM转换<br>3. 教学清晰：学员可直接查看数据结构和处理流程<br>4. 溯源简单：通过索引直接定位原文，无需相似度计算<br>**后果**：无法支持语义搜索；大量研报时检索性能下降<br>**替代方案**：向量数据库（P2考虑）：支持语义搜索但引入额外复杂度；Elasticsearch：支持全文检索但部署复杂 |

## 8. 安全架构

> 对齐 `03` §5 风险和 `04` 产品规则。当前教学版与生产需求的差距。

| 层 | 当前措施 | 生产改进 |
|----|----------|----------|
| 认证 | 登录界面模拟，无真实鉴权 | JWT Token + 用户密码验证 |
| 角色权限 | 前端角色选择，无后端校验 | RBAC权限控制，投资经理/分析师权限区分 |
| CORS | 开发环境全开放 | 生产白名单限制 |
| 密钥 | 环境变量 `.env` | Vault / Secrets Manager |
| 文件上传 | 仅校验文件扩展名 | 文件类型魔数校验 + 病毒扫描 + 大小限制（50MB） |
| 输入校验 | 基础长度校验（1-500字符） | 完整的XSS防护 + SQL注入防护 + 参数化查询 |
| 审计日志 | 无 | 完整的操作日志记录与留存，支持合规审计 |
| 访问控制 | 无 | 用户只能访问自己的会话和问答记录 |
| 传输安全 | HTTP明文传输 | HTTPS/TLS加密传输 |
| 数据加密 | 无 | 敏感数据（API密钥、用户数据）加密存储 |
| 限流熔断 | 无 | API限流防止滥用，LLM调用熔断保护 |
| 溯源真实性 | Demo模式返回模拟溯源 | 生产环境必须真实关联原文段落 |
