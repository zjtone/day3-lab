## 5. 依赖与风险

**依赖链**：

```
S1 会话管理基础 → S2 问答核心与降级 → S3 历史记录与导出 → S4 能力状态与优化
   (W1,W2,W4)        (W1,W3,W4,W5)        (W1,W2,W4)          (W1,W3,W4,W6)
```

**关键路径**：

| 前置条件 | 说明 |
|----------|------|
| S1 完成前 | W2.1 Storage 架构必须先于 W4.2 Sidebar 完成（数据层先于展示层） |
| S1 完成前 | W1.1 路由基础必须先于 W1.2 会话路由完成 |
| S2 完成前 | W2.2 Session 存储必须先于 W2.3 QARecord 存储完成 |
| S2 完成前 | W5.1/W5.2 Provider 层必须先于 W3.2 三级降级完成 |
| S2 完成前 | W3.2 三级降级必须先于 W3.3 流式响应完成 |
| S3 完成前 | W2.3 QARecord 存储必须先于 W2.4 导出存储完成 |
| S4 完成前 | W1.3 问答路由必须先于 W1.5 能力探测路由完成 |

**外部依赖**：

| 依赖项 | 影响阶段 | 说明 |
|--------|----------|------|
| CoPaw API | S2 | 需要可用的 CoPaw 服务地址和认证信息 |
| 百炼 DashScope API Key | S2 | 需要有效的百炼 API Key |
| Python 3.8+ | S1 | 后端运行环境 |
| Node.js 16+ | S1 | 前端运行环境 |

| 风险 | 影响 | 应对 |
|------|------|------|
| 上游 LLM 不可用 | 问答功能降级，用户体验下降 | Demo 模式兜底，确保基础功能可用（REQ-M1QA-006） |
| 外部 API 响应超时 | 用户等待时间过长，体验差 | 设置 20s/120s 分级超时，超时自动降级 |
| JSON 文件并发写入冲突 | 数据丢失或损坏 | Storage 层加文件锁，或切换至 SQLite |
| 前端状态管理复杂 | 流式响应状态同步问题 | 使用 React Hooks 严格管理，增加单元测试 |
| 开发时间受限 | 约 3 小时开发时间，功能需简化 | 采用 AI 生成模拟数据，聚焦核心功能 |
| 研报解析格式复杂 | PDF/Word 格式多样，解析失败率高 | 优先支持常见格式，异常文件人工处理 |
